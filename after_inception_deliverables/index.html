Architecture Principles · DGTP-ARCH-001

15 Governing Principles

Every design decision across all 10 deliverables must satisfy the applicable principles. Non-negotiable — waivers require a formal Architecture Decision Record.

Four Principle Domains
Principles are grouped into four domains. Each deliverable cites the principles that govern it. When a principle and a practical constraint conflict, the principle takes precedence.
⚙ Platform — P-01 to P-06
API design, cloud platform, layer separation, coupling, resilience, open standards.
🔒 Security — P-07 to P-09
Security by design, identity-first access, privacy and data protection.
💾 Data — P-10 to P-12
Authoritative source, once-only principle, data sovereignty and residency.
📋 Governance — P-13 to P-15
Observability, infrastructure as code, compliance by design.
⚙ Platform — P-01 to P-06
P-01 API-First All APIs documented in OpenAPI 3.0 and agreed by the Architecture Team before development begins. GoSXM API Catalogue is the single source of truth.
P-02 Cloud-Native Azure Exclusive Azure PaaS in European region. No multi-cloud or on-premise in Phase 1. All deployments via IaC — no manual portal changes in staging or production.
P-03 Separation of Concerns Seven enforced platform layers. Each has one responsibility. No cross-layer shortcutting. Experience Layer cannot query registries directly — ever.
P-04 Loose Coupling One dedicated, independently deployable adapter per registry. A registry changing its protocol affects only its adapter — zero platform-wide impact.
P-05 Resilience by Default Outbox Pattern in all adapters guarantees zero data loss. Service Bus zone-redundant with Geo-DR. Circuit breakers prevent cascade failures.
P-06 Open Standards REST/JSON, OpenAPI 3.0, OAuth2/OIDC, OData where available. Vendor lock-in minimised. All interface contracts machine-readable and publicly documented.
🔒 Security — P-07 to P-09
P-07 Security by Design Security embedded at every layer from Day 1. mTLS between all services. OWASP review is a mandatory go-live gate. SAST/CVE scanning in all CI/CD pipelines.
P-08 Identity-First Access Azure Entra External ID is the sole identity provider. 7 RBAC roles. JWT 15-min TTL. PIM Just-In-Time for all privileged access. No standing admin privileges.
P-09 Privacy by Design Data minimisation and consent management built into every cross-registry call. Citizens can view their data access history. 90-day PII audit log.
💾 Data — P-10 to P-12
P-10 Authoritative Source Each data domain has exactly one authoritative registry. Canonical Data Store is a read-model — it never becomes the source of truth. Registries own their data.
P-11 Once-Only Principle Citizens submit data once. The platform reuses it across services via Canonical Data Store with a freshness SLA of less than 15 minutes.
P-12 Data Sovereignty All citizen data stays within Azure European region at all times. Geo-DR secondary must also be European. No third-party SaaS without a signed EU data processing agreement.
📋 Governance — P-13 to P-15
P-13 Observability First Structured logging, distributed tracing, and metrics from Day 1. Azure Monitor + Application Insights. Alerts on DLQ first-message within 2 minutes.
P-14 Infrastructure as Code All Azure resources provisioned via Bicep/Terraform. No manual portal changes in staging or production. IaC enforces region and security constraints in every template.
P-15 Compliance by Design ID-Exchange EA Ordinance approval is a hard go-live gate. Privacy Ordinance compliance built into data flows. Audit logs satisfy statutory reporting requirements.

Q1
DATA GOVERNANCE
Who owns the Canonical Data Store — DGTP or each registry?
A citizen updates their address at Civil Registry. The canonical store is stale 12 minutes. Who authorises a correction?
BLOCKS BUILD PHASE↗
Q2
IDENTITY & ACCESS
Will GovPass/eID be available for all citizens on Day 1 of go-live?
If 30% of Sint Maarten citizens lack a GovPass account, they cannot access any government service through DGTP on launch day.
BLOCKS BUILD PHASE↗
Q3
LEGAL & COMPLIANCE
Has the EA Ordinance been approved — and does it permit EU-region Azure hosting?
All DGTP infrastructure is in Azure West Europe. If the EA Ordinance requires Kingdom of Netherlands hosting specifically, every IaC template in D8 and D9 must be redesigned.
BLOCKS BUILD PHASE↗
Q4
REGISTRY READINESS
Which registries have stable sandbox API endpoints for Build Phase Sprint 1?
CASBES and COCI adapters are scheduled for Sprint 1. If CASBES can't provide a stable StUF-BG SOAP endpoint by Sprint 1 Week 1, the most critical integration can't be built. Sprint 1 delivers nothing testable.
BLOCKS BUILD PHASE↗
Q5
ONCE-ONLY PRINCIPLE
Has SXM formally adopted P-11 Once-Only for citizen data sharing?
Without formal policy adoption, every cross-registry data reuse requires a new citizen consent popup — making every citizen transaction 3× slower and dramatically increasing Consent Service complexity in D3 and D8.
BLOCKS BUILD PHASE↗
Q6
PROJECT GOVERNANCE
Who is the SXM Government Architecture counterpart with sign-off authority?
D8 is the Build Phase gate document requiring Architecture Team sign-off. Every ADR in the Build Phase also requires AT approval. Without a named SXM counterpart, every gate document becomes a bureaucratic bottleneck.
BLOCKS BUILD PHASE↗
AD1
ARCHITECTURE DECISION
Should APIM be the single gateway for ALL government APIs, or only DGTP registries?
SXM Government has other existing APIs (tax payment, permit submission, customs). If not onboarded to APIM, citizens must authenticate separately to each system — breaking the once-only login experience.
BLOCKS BUILD PHASE↗
AD2
ARCHITECTURE DECISION
Is the Canonical Data Store the system of record — or a read cache only?
If a government officer uses a canonical store value (showing licence ACTIVE) to grant access, but the true COCI value is SUSPENDED, the decision is legally challengeable — and DGTP is in the evidence chain.
BLOCKS BUILD PHASE↗
AD3
ARCHITECTURE DECISION
Who manages APIM policy changes in Build Phase and in production?
During a security incident, an emergency WAF rule update is needed. Who can approve and deploy it — and in how many minutes? Without a defined process, exposure window extends until business hours resume.
BLOCKS BUILD PHASE↗
AD4
ARCHITECTURE DECISION
Who declares the platform ready for go-live — and what is the minimum criteria?
Build Phase completes. Staging smoke tests pass. But the Civil Registry focal point says they need 2 more weeks of parallel running. The programme manager wants to go live. The Architecture Team says wait. Who decides?
BLOCKS BUILD PHASE↗
