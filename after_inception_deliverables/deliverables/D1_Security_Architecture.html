7 RBAC ROLES — P-08 Identity-First Access
Citizen OAuth2 PKCE + eID / GovPass ▾
Own applications, document wallet, and status tracking only. NSN (national ID number) in JWT must match the application being accessed — Attribute-Based Access Control enforced at the Backend-for-Frontend layer. Per P-09 Privacy by Design, citizens must be able to view what data the platform holds about them and which services accessed it.
Government Staff SSO via Entra ID Federation ▾
Cross-registry lookup, case processing, and citizen verification. SSO enabled across all platform portals via Entra External ID federation — one login for all portals. All access events logged with officer identity, timestamp, and stated purpose per P-09 Privacy by Design compliance requirements.
Registry Admin PIM Just-In-Time Only ▾
Registry-scoped administration only — cannot access other registries' canonical data without explicit authorisation per P-09 Privacy by Design data minimisation. Every PIM activation requires MFA and a documented business justification. No standing admin privileges.
Platform Admin · Integration Admin · Auditor · Read-Only ▾
Platform Admin: Full platform administration — PIM JIT only. All actions WORM-logged via Sentinel. No standing access.
Integration Admin: Adapter config, Service Bus topics, DLQ inspection. PIM-gated.
Auditor: Read-only to all audit logs and compliance reports (90-day window).
Read-Only Viewer: Canonical Data Store reads within authorised scope only.

🔒 Encryption Controls▾
TLS 1.2 minimum everywhere (TLS 1.3 preferred for new connections). AES-256 via Azure Storage Service Encryption. Customer-managed key (CMK) upgrade path available. All keys and certificates in Azure Key Vault — no secrets in source code, config files, or environment variables at any time.
📋 PII Minimisation & Consent (P-09 Privacy by Design)▾
Only the minimum data required for a specific transaction may be retrieved from any registry. All cross-agency data reuse tracked with consent record and documented legal basis. Every PII access logged: who, what, when, purpose, legal basis. 90-day minimum retention. Withdrawal of consent triggers automated deletion workflows per P-09 Privacy by Design.
🌍 Data Residency (P-12 Data Sovereignty)▾
All citizen data must remain within the Azure European region at all times. IaC (Bicep/Terraform) region constraints enforced in all templates per P-14 Infrastructure as Code Infrastructure as Code. Service Bus Geo-DR secondary must also be European. No third-party SaaS without a signed EU data processing agreement.

⚙ ARCHITECTURE PRINCIPLES — P-01 to P-15 · Click any row to expand description
P-01
API-First
All platform APIs documented in OpenAPI 3.0 and agreed by Architecture Team BEFORE development begins. GoSXM API Catalogue is the single source of truth.
▾
P-02
Cloud-Native Azure
Exclusive Azure PaaS in European region. No multi-cloud, no on-premise in Phase 1. All deployments via IaC — no manual portal changes in staging or production.
▾
P-03
Separation of Concerns
Seven enforced platform layers. Each layer has one responsibility. No cross-layer shortcutting. Experience Layer cannot query registries directly.
▾
P-04
Loose Coupling
One dedicated, independently deployable adapter per registry. A registry changing its protocol affects only its adapter — zero platform-wide impact.
▾
P-05
Resilience by Default
Outbox Pattern in all adapters guarantees zero data loss. Service Bus zone-redundant with Geo-DR. Circuit breakers prevent cascade failures.
▾
P-06
Open Standards
REST/JSON, OpenAPI 3.0, OAuth2/OIDC, OData where available. Vendor lock-in minimised. All interface contracts machine-readable and publicly documented.
▾
P-07
Security by Design
Security embedded at every layer from Day 1. mTLS between all services. OWASP review is a go-live gate. SAST/CVE scanning in all CI/CD pipelines.
▾
P-08
Identity-First Access
Azure Entra External ID is the sole identity provider. 7 RBAC roles. JWT 15-min TTL. PIM Just-In-Time for all privileged access. No standing admin privileges.
▾
P-09
Privacy by Design
Data minimisation and consent management built into every cross-registry call. Citizens can view their data access history. 90-day PII audit log in Log Analytics.
▾
P-10
Authoritative Source
Each data domain has exactly one authoritative registry. Canonical Data Store is a read-model — it never becomes the source of truth. Registries own their data.
▾
P-11
Once-Only Principle
Citizens submit data once. The platform reuses it across services via the Canonical Data Store with a freshness SLA of less than 15 minutes. No re-entry of known data.
▾
P-12
Data Sovereignty
All citizen data stays within Azure European region at all times. Geo-DR secondary must also be European. No third-party SaaS without a signed EU data processing agreement.
▾
P-13
Observability First
Structured logging, distributed tracing, and metrics from Day 1 in every component. Azure Monitor + Application Insights. Alerts on DLQ first-message within 2 minutes.
▾
P-14
Infrastructure as Code
All Azure resources provisioned via Bicep/Terraform. No manual portal changes in staging or production. IaC enforces region and security constraints in every template.
▾
P-15
Compliance by Design
ID-Exchange EA Ordinance approval is a hard go-live gate. Privacy Ordinance compliance built into data flows. Audit logs satisfy statutory reporting requirements.
▾
