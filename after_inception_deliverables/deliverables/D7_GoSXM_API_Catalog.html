The single source of truth for all platform APIs. Azure APIM Developer Portal with Swagger UI · try-it-out capability · role-gated consumer access · CI/CD auto-publication from approved OpenAPI 3.0 specs. Every approved API is visible. Nothing is undocumented.

P-01 API-First
API-First — Catalogue as Source of Truth
GoSXM API Catalogue is the authoritative reference for every API published on the platform. All approved OpenAPI 3.0 specs are published here automatically. No API is available to consumers unless it appears in the Catalogue. Every spec change triggers a CI/CD pipeline that updates the Catalogue within minutes.

CATALOGUE RULES — P-01 API-First
No API is available without a Catalogue entry▾
Per P-01 API-First API-First, every platform API must have an approved OpenAPI 3.0 spec published in the GoSXM API Catalogue before any consumer can access it. A deployed APIM endpoint that is not in the Catalogue is not a published API — it is an undocumented endpoint and must be removed or catalogued immediately.
No manual catalogue updates ever▾
The Catalogue is updated exclusively via CI/CD pipeline triggered by an approved API deployment. No team member may manually edit Catalogue entries, update spec files directly in APIM, or add endpoints outside the pipeline. Manual updates create drift between deployed APIM and the Catalogue — violating P-01 API-First.
Subscription self-service with admin approval▾
Consumers browse the Catalogue and self-select API subscriptions. All subscription requests require Integration Admin approval before access is granted. Approval decisions are logged with timestamp, approver, and business justification. Subscription scope is limited to the consumer's RBAC role — no consumer can subscribe to endpoints outside their role.
Quarterly access review for all subscriptions▾
All active API subscriptions are reviewed quarterly by the Integration Admin. Subscriptions with no API calls in the previous 90 days are revoked after written notice to the consumer. External developer subscriptions are reviewed monthly — not quarterly. All reviews are logged in the audit trail per P-13 Observability First Observability First.
Design Decision — P-01 API-First
Should the GoSXM API Catalogue be publicly accessible (no auth to browse) or gated (Entra login required to view)?
Public: easier developer discovery, aligns with open government principles. Gated: prevents enumeration of government API surface by unauthorised parties. Architecture Team decision required — it affects the Entra configuration and the APIM developer portal settings.

D7 · P-01 API-First · P-13 Observability First CI/CD Publication Pipeline & Document Map

CI/CD Publication Pipeline & Document Map

Every approved spec deployment triggers an automatic Catalogue update. No manual steps. The Catalogue is always in sync with what is deployed in Azure APIM — guaranteed by the pipeline, not by discipline.

CI/CD PIPELINE — SPEC TO CATALOGUE
1
Architecture Team approves OpenAPI spec
Spec review complete, registry validation done, sign-off received. Spec PR merged to the central spec repository.
2
GitHub Actions pipeline triggered on merge
Pipeline validates spec (OpenAPI linting), imports to Azure APIM via ARM/Bicep, updates APIM Developer Portal, and creates/updates the Catalogue entry.
3
Catalogue updated within minutes
GoSXM API Catalogue reflects the new spec. Existing subscriptions remain active. All consumers see the updated documentation immediately.
✓
Azure Monitor alert confirms successful publication
Per P-13 Observability First Observability First — every Catalogue update triggers a confirmation alert to the Integration Admin. Failed pipeline runs trigger a P1 alert within 2 minutes.
Breaking change protection: If a deployed spec is a breaking change (/v2/ promotion), the pipeline automatically sends deprecation notices to all active subscribers of the /v1/ endpoint with the 4-week notice period defined in D4 API Design Guidelines.

Catalogue Monitoring & API Governance Metrics

Per P-13 Observability First Observability First — the Catalogue is not just a documentation portal. It is an active governance instrument. API usage, subscription velocity, error rates, and deprecation status are tracked and reported to the Architecture Team weekly.

CATALOGUE GOVERNANCE METRICS — P-13 Observability First

CATALOGUE FRESHNESS
<5 min

SUBSCRIPTION APPROVAL
<2 days

STALE SUBSCRIPTION RATE
<5%
Target: fewer than 5% of active subscriptions with zero API calls in the previous 90 days — enforced by quarterly review
Weekly Architecture Team Report▾
Every Monday: Integration Admin produces a Catalogue health report covering: new subscriptions (approved/rejected), top 5 most-called APIs, APIs with zero calls in 30 days, deprecation notices sent, pipeline failures in the past week, and any consumer complaints about spec accuracy. Distributed to Architecture Team and Project Steering Committee.
Deprecation Tracking Dashboard▾
All /v1/ endpoints with a planned /v2/ promotion are tracked in the Catalogue with a deprecation countdown banner visible to all subscribers. When the 4-week notice period begins, all active subscribers receive an automated email. The Catalogue shows which consumers have migrated to /v2/ and which are still on the deprecated /v1/ endpoint — enabling the Architecture Team to follow up.

⚙ ARCHITECTURE PRINCIPLES — P-01 to P-15 · Click any row to expand description
P-01
API-First
All platform APIs documented in OpenAPI 3.0 and agreed by Architecture Team BEFORE development begins. GoSXM API Catalogue is the single source of truth.
▾
P-02
Cloud-Native Azure
Exclusive Azure PaaS in European region. No multi-cloud, no on-premise in Phase 1. All deployments via IaC — no manual portal changes in staging or production.
▾
P-03
Separation of Concerns
Seven enforced platform layers. Each layer has one responsibility. No cross-layer shortcutting. Experience Layer cannot query registries directly.
▾
P-04
Loose Coupling
One dedicated, independently deployable adapter per registry. A registry changing its protocol affects only its adapter — zero platform-wide impact.
▾
P-05
Resilience by Default
Outbox Pattern in all adapters guarantees zero data loss. Service Bus zone-redundant with Geo-DR. Circuit breakers prevent cascade failures.
▾
P-06
Open Standards
REST/JSON, OpenAPI 3.0, OAuth2/OIDC, OData where available. Vendor lock-in minimised. All interface contracts machine-readable and publicly documented.
▾
P-07
Security by Design
Security embedded at every layer from Day 1. mTLS between all services. OWASP review is a go-live gate. SAST/CVE scanning in all CI/CD pipelines.
▾
P-08
Identity-First Access
Azure Entra External ID is the sole identity provider. 7 RBAC roles. JWT 15-min TTL. PIM Just-In-Time for all privileged access. No standing admin privileges.
▾
P-09
Privacy by Design
Data minimisation and consent management built into every cross-registry call. Citizens can view their data access history. 90-day PII audit log in Log Analytics.
▾
P-10
Authoritative Source
Each data domain has exactly one authoritative registry. Canonical Data Store is a read-model — it never becomes the source of truth. Registries own their data.
▾
P-11
Once-Only Principle
Citizens submit data once. The platform reuses it across services via the Canonical Data Store with a freshness SLA of less than 15 minutes. No re-entry of known data.
▾
P-12
Data Sovereignty
All citizen data stays within Azure European region at all times. Geo-DR secondary must also be European. No third-party SaaS without a signed EU data processing agreement.
▾
P-13
Observability First
Structured logging, distributed tracing, and metrics from Day 1 in every component. Azure Monitor + Application Insights. Alerts on DLQ first-message within 2 minutes.
▾
P-14
Infrastructure as Code
All Azure resources provisioned via Bicep/Terraform. No manual portal changes in staging or production. IaC enforces region and security constraints in every template.
▾
P-15
Compliance by Design
ID-Exchange EA Ordinance approval is a hard go-live gate. Privacy Ordinance compliance built into data flows. Audit logs satisfy statutory reporting requirements.
▾
