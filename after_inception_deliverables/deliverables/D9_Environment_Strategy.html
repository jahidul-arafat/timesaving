Environment Strategy

Three-environment topology: DEV · Staging · Production. All provisioned via IaC — no manual Azure portal configurations ever. European region enforced by Azure Policy. CI/CD pipelines govern all deployments from development through to production.

P-14 Infrastructure as Code
Infrastructure as Code — Everything
Every Azure resource in every environment is provisioned by a Bicep or Terraform template stored in source control. No manual portal provisioning ever — not in DEV, not in staging, not in production. Any manual change is detected by drift detection and flagged as a policy violation within 15 minutes.
ZERO MANUAL PROVISIONING
P-02 Cloud-Native Azure
Cloud-Native Azure — EU Region Only
All three environments are provisioned exclusively in Azure European regions — West Europe (primary) and North Europe (secondary for Geo-DR). Azure Policy blocks any resource creation outside these regions at the subscription level. No exceptions under any circumstance, including cost optimisation.
EU REGION ENFORCED
P-13 Observability First
Observability First — All Environments
Structured logging, distributed tracing, and metrics are enabled in all three environments from Day 1 — not just production. DEV has Application Insights. Staging has full Azure Monitor. Production has all monitoring plus SIEM integration. Alerts are active in staging before production go-live.

Staging — Production Replica Rules▾
Staging must be an exact replica of production architecture — same Azure resource types, same SKUs, same network topology, same IaC templates with environment-specific parameter files. If staging differs from production, smoke tests cannot be trusted. Staging uses mock data for all 6 registries — no live citizen data ever. Auto-refreshed weekly. Used for D5 registry sandbox sessions and D6 validation.
Production Resource Group Topology▾
rg-dgtp-prod-core: AKS, APIM, Entra configurations, Azure Front Door + WAF.
rg-dgtp-prod-data: Cosmos DB, Azure SQL, Key Vault (data), Log Analytics Workspace.
rg-dgtp-prod-integration: Service Bus, Container Apps (adapters), Key Vault (adapters).
rg-dgtp-prod-monitoring: Azure Monitor, Application Insights, Alert rules, Workbooks.
rg-dgtp-prod-security: Microsoft Sentinel (SIEM), Defender for Cloud, DDoS Protection.
rg-dgtp-prod-operations: Automation Accounts, Runbooks, Cost Management.
DEV Environment Cost Controls▾
DEV environments are cost-optimised: Container Apps namespaces auto-scale to zero when idle. APIM in Consumption tier. Cosmos DB in serverless mode. Azure Automation runbooks shut down all DEV resources at 18:00 CET daily and restart at 08:00 CET. Developers can opt out for overnight builds. Monthly DEV cost target: under €500 total.

D9 · P-14 Infrastructure as Code · P-07 Security by Design CI/CD Pipeline Design — DEV → Staging → Production

CI/CD Pipeline Design

Every deployment flows through the CI/CD pipeline. No direct pushes to staging or production. No manual deployments. Architecture Team approval is required before any production deployment. SAST gates block Critical and High CVEs.

PIPELINE STAGES — GITHUB ACTIONS
# On: push to feature branch → PR → merge to main
jobs:
build:
- checkout + install deps
- dotnet build / npm ci
- unit tests (coverage >80%)

security-scan:
- SAST: CodeQL analysis
- Dependency: OWASP Dep-Check
- Container: Trivy scan
FAIL on Critical or High CVE

iac-validate:
- az deployment what-if
- bicep lint + openapi lint
- policy compliance check

deploy-staging: # auto on main merge
- az deployment create (staging)
- smoke tests (end-to-end)
- performance baseline check

deploy-production: # AT approval required
- manual approval gate (AT)
- az deployment create (prod)
- post-deploy validation
- alert: Monitor + Sentinel
- rollback trigger on failure
PIPELINE GOVERNANCE RULES
SAST gate — fail on Critical or High CVE▾
Every pipeline run performs: CodeQL static analysis (code vulnerabilities), OWASP Dependency Check (library CVEs), and Trivy container scan (image vulnerabilities). Any Critical or High severity finding fails the pipeline. The deployment cannot proceed until the finding is remediated or an Architecture Team-approved exception is recorded as an ADR. No exceptions by individual developers.
Branch protection — no direct production pushes▾
GitHub branch protection rules: main branch requires 2 reviewer approvals before merge. Staging branch auto-deploys on merge to main. Production branch requires Architecture Team manual approval gate in GitHub Actions (environment protection rule). Force pushes to main, staging, or production branches are disabled. All PRs must pass all pipeline stages before merge.
Rollback procedure — automated trigger▾
If a production deployment fails post-deploy validation (smoke tests, health checks, error rate spike in Azure Monitor), the pipeline triggers an automatic rollback to the previous version via az containerapp revision activate. RTO for rollback: under 5 minutes. Rollback trigger thresholds: error rate >5% in 2 minutes, or P95 latency >10s, or any health check failure.
IaC drift detection — 15-minute alert▾
Azure Policy combined with a scheduled pipeline job runs az deployment what-if against all production resource groups every 15 minutes. Any drift (manual portal change, unexpected resource modification) triggers a P1 alert to the Integration Admin and Architecture Team. The alert includes the diff of what changed and who made the change (via Azure Activity Log). The drift must be remediated within 4 hours.
Design Decision
Should the production deployment pipeline require one or two Architecture Team approvals?
Single approval: faster deployments. Two approvals: stronger separation of duties — aligns with P-07 Security by Design Security by Design and government change management standards. Recommendation: two approvals for production. Architecture Team must confirm in D8 Section 6 (Deployment Runbook).

Monitoring & Alerting Specification

Azure Monitor + Application Insights▾
All platform components emit structured logs (JSON) to a single Log Analytics Workspace. Application Insights provides distributed tracing — every request gets a correlation ID that travels through APIM → Business Services → Integration Layer → Adapter. Trace ID included in all error responses so consumers can report and the team can find the exact request chain in seconds.
Key Alert Rules — P1 (immediate) vs P2 (15 min)▾
P1 — immediate, on-call page:
• DLQ first-message: Service Bus DLQ receives any message → 2-minute alert
• Health check failure: any component health endpoint fails → immediate
• Error rate >5% over 2 minutes → production rollback trigger
• IaC drift detected in production → immediate

P2 — 15-minute response:
• Canonical freshness lag >12 minutes (SLA warning before breach)
• P95 API latency >1.5s (warning before 2s SLA breach)
• Rate limit approaching platform total (4,500 req/min of 5,000 ceiling)
• Certificate expiry <30 days remaining
Microsoft Sentinel SIEM (Production)▾
Production environment integrates with Microsoft Sentinel for Security Information and Event Management. Sentinel analytics rules: failed authentication spike, unusual API call patterns (volume, geography), PIM activation outside business hours, IaC drift, DDoS threshold exceeded. All P1 security alerts escalate to the designated security contact immediately. Sentinel retains 90-day log history per P-09 Privacy by Design Privacy by Design audit requirements.
OBSERVABILITY PER ENVIRONMENT
DEV Application Insights · no alerts
Distributed tracing and logs only. No alert rules — developers review logs manually. Application Insights SDK in all components from Day 1. Trace ID in all responses.
STAGING Full Azure Monitor · P2 alerts to team
All P1 and P2 alert rules active — but no on-call paging. Alerts go to team Slack/email. Used to tune alert thresholds before production go-live. DLQ monitoring active. IaC drift detection active.
PRODUCTION Full Monitor + Sentinel · on-call paging
All P1 alerts trigger on-call page via Azure Monitor Action Groups (PagerDuty or Azure native). P2 alerts to team channel. Sentinel SIEM active. 90-day log retention. All citizen data access logged per P-09 Privacy by Design.
Alert tuning in staging is mandatory: All production alert thresholds must be validated in staging under realistic load (200 concurrent users, all 6 adapter mock responses) before production go-live. Untested alerts in production create false positives that desensitise the on-call team.

⚙ ARCHITECTURE PRINCIPLES — P-01 to P-15 · Click any row to expand description
P-01
API-First
All platform APIs documented in OpenAPI 3.0 and agreed by Architecture Team BEFORE development begins. GoSXM API Catalogue is the single source of truth.
▾
P-02
Cloud-Native Azure
Exclusive Azure PaaS in European region. No multi-cloud, no on-premise in Phase 1. All deployments via IaC — no manual portal changes in staging or production.
▾
P-03
Separation of Concerns
Seven enforced platform layers. Each layer has one responsibility. No cross-layer shortcutting. Experience Layer cannot query registries directly.
▾
P-04
Loose Coupling
One dedicated, independently deployable adapter per registry. A registry changing its protocol affects only its adapter — zero platform-wide impact.
▾
P-05
Resilience by Default
Outbox Pattern in all adapters guarantees zero data loss. Service Bus zone-redundant with Geo-DR. Circuit breakers prevent cascade failures.
▾
P-06
Open Standards
REST/JSON, OpenAPI 3.0, OAuth2/OIDC, OData where available. Vendor lock-in minimised. All interface contracts machine-readable and publicly documented.
▾
P-07
Security by Design
Security embedded at every layer from Day 1. mTLS between all services. OWASP review is a go-live gate. SAST/CVE scanning in all CI/CD pipelines.
▾
P-08
Identity-First Access
Azure Entra External ID is the sole identity provider. 7 RBAC roles. JWT 15-min TTL. PIM Just-In-Time for all privileged access. No standing admin privileges.
▾
P-09
Privacy by Design
Data minimisation and consent management built into every cross-registry call. Citizens can view their data access history. 90-day PII audit log in Log Analytics.
▾
P-10
Authoritative Source
Each data domain has exactly one authoritative registry. Canonical Data Store is a read-model — it never becomes the source of truth. Registries own their data.
▾
P-11
Once-Only Principle
Citizens submit data once. The platform reuses it across services via the Canonical Data Store with a freshness SLA of less than 15 minutes. No re-entry of known data.
▾
P-12
Data Sovereignty
All citizen data stays within Azure European region at all times. Geo-DR secondary must also be European. No third-party SaaS without a signed EU data processing agreement.
▾
P-13
Observability First
Structured logging, distributed tracing, and metrics from Day 1 in every component. Azure Monitor + Application Insights. Alerts on DLQ first-message within 2 minutes.
▾
P-14
Infrastructure as Code
All Azure resources provisioned via Bicep/Terraform. No manual portal changes in staging or production. IaC enforces region and security constraints in every template.
▾
P-15
Compliance by Design
ID-Exchange EA Ordinance approval is a hard go-live gate. Privacy Ordinance compliance built into data flows. Audit logs satisfy statutory reporting requirements.
▾
