D4 · P-01 API-First · P-06 Open Standards Per-Registry API Design — Protocol to OpenAPI 3.0

Six Registry API Design Challenges

Each registry speaks a different protocol. Each adapter must translate it into a conformant OpenAPI 3.0 spec agreed by the Architecture Team — before any code is written. Hover the flow diagram for details.

REGISTRY PROTOCOL → ADAPTER → OPENAPI 3.0 → GOSXM CATALOGUE · HOVER EACH NODE
PER-REGISTRY SPEC CHALLENGES — P-01 API-First
Civil Registry — StUF-BG SOAP → REST▾
APIM SOAP policy transforms StUF-BG XML to REST/JSON. Namespace versioning requires field-by-field mapping. NSN (national ID) is the primary citizen key — all registries cross-reference it. OpenAPI spec must document every field mapping exhaustively. WSDL and schema files must be secured from the Civil Registry technical contact before spec writing begins.
COCI — Oracle OData v4 → OpenAPI 3.0▾
OData v4 facade wrapped in OpenAPI 3.0. CRIB number cross-reference with Tax Administration is a required field linkage. Director/shareholder nested entity structure requires careful JSON Schema $ref design. Documentation is available from the COCI focal point.
Tax SIAH — MS Dynamics CRM ⚠ Access Gated▾
MS CRM internal API wrapped as REST. Restricted external access must be resolved via data sharing agreement BEFORE spec design can begin. CRIB number cross-references COCI. Spec design formally gated on Project Steering Committee approval.
TEATT — D365 OData v4▾
D365 OData v4 endpoint. Licence status enum (ACTIVE / SUSPENDED / EXPIRED / PENDING) and all business rules governing transitions must be in the OpenAPI spec — not left to adapter implementation. Holder NSN cross-references Civil Registry.
VROMI — ArcGIS REST Feature Service▾
ArcGIS Feature Service REST. WKT (Well-Known Text) geometry for parcel boundaries in response schema. Parcel ID spatial queries. Best As-Is documentation — begin this spec in parallel with COCI to maximise delivery velocity.
Design Decision — P-01 API-First
Should OpenAPI specs live inside each adapter repo or in a dedicated central API spec repo?
Central: single source of truth, easier governance, one CI/CD pipeline to APIM. Per-adapter: spec and implementation co-located. Decision must be made before any adapter development begins — it affects the entire CI/CD and Catalogue publishing workflow.

Open Question — P-01 API-First
Who holds final authority to approve a spec — Architecture Team alone, or joint approval with each registry technical contact?
P-01 API-First requires Architecture Team approval. But registry contacts must validate system accuracy. A joint governance process must be defined before any spec is submitted for review.

⚙ ARCHITECTURE PRINCIPLES — P-01 to P-15 · Click any row to expand description
P-01
API-First
All platform APIs documented in OpenAPI 3.0 and agreed by Architecture Team BEFORE development begins. GoSXM API Catalogue is the single source of truth.
▾
P-02
Cloud-Native Azure
Exclusive Azure PaaS in European region. No multi-cloud, no on-premise in Phase 1. All deployments via IaC — no manual portal changes in staging or production.
▾
P-03
Separation of Concerns
Seven enforced platform layers. Each layer has one responsibility. No cross-layer shortcutting. Experience Layer cannot query registries directly.
▾
P-04
Loose Coupling
One dedicated, independently deployable adapter per registry. A registry changing its protocol affects only its adapter — zero platform-wide impact.
▾
P-05
Resilience by Default
Outbox Pattern in all adapters guarantees zero data loss. Service Bus zone-redundant with Geo-DR. Circuit breakers prevent cascade failures.
▾
P-06
Open Standards
REST/JSON, OpenAPI 3.0, OAuth2/OIDC, OData where available. Vendor lock-in minimised. All interface contracts machine-readable and publicly documented.
▾
P-07
Security by Design
Security embedded at every layer from Day 1. mTLS between all services. OWASP review is a go-live gate. SAST/CVE scanning in all CI/CD pipelines.
▾
P-08
Identity-First Access
Azure Entra External ID is the sole identity provider. 7 RBAC roles. JWT 15-min TTL. PIM Just-In-Time for all privileged access. No standing admin privileges.
▾
P-09
Privacy by Design
Data minimisation and consent management built into every cross-registry call. Citizens can view their data access history. 90-day PII audit log in Log Analytics.
▾
P-10
Authoritative Source
Each data domain has exactly one authoritative registry. Canonical Data Store is a read-model — it never becomes the source of truth. Registries own their data.
▾
P-11
Once-Only Principle
Citizens submit data once. The platform reuses it across services via the Canonical Data Store with a freshness SLA of less than 15 minutes. No re-entry of known data.
▾
P-12
Data Sovereignty
All citizen data stays within Azure European region at all times. Geo-DR secondary must also be European. No third-party SaaS without a signed EU data processing agreement.
▾
P-13
Observability First
Structured logging, distributed tracing, and metrics from Day 1 in every component. Azure Monitor + Application Insights. Alerts on DLQ first-message within 2 minutes.
▾
P-14
Infrastructure as Code
All Azure resources provisioned via Bicep/Terraform. No manual portal changes in staging or production. IaC enforces region and security constraints in every template.
▾
P-15
Compliance by Design
ID-Exchange EA Ordinance approval is a hard go-live gate. Privacy Ordinance compliance built into data flows. Audit logs satisfy statutory reporting requirements.
▾
