D6 · Per-Registry Support Plan Risk-Rated Support Approach per Registry

Per-Registry Support Plan & Risk Ratings

Support intensity and approach varies by registry risk rating. High-risk registries get weekly sessions. Lower-risk registries get bi-weekly. All have a dedicated DGTP point of contact from Day 1 of D5 delivery.

🔴 HIGH RISK · Weekly sessions
Civil Registry (CASBES)
Protocol: StUF-BG SOAP — highest adapter complexity
Risk: Knowledge concentration — few people hold all institutional knowledge
Session focus: SOAP schema mapping, WSDL validation, NSN field edge cases, APIM transformation policy testing
Action: Retain copies of all docs after every session. StUF-BG specialist required from DGTP side
🟡 MEDIUM RISK · Weekly sessions
Tax Administration (SIAH)
Protocol: MS Dynamics CRM — restricted external access
Risk: Data sharing agreement not yet signed — sessions are pre-integration planning only until DSA is signed
Session focus: DSA negotiation support, CRM API capability review, CRIB cross-reference validation
Action: Escalate to PSC if DSA not signed within 6 weeks of D5 delivery
🟡 MEDIUM RISK · Bi-weekly sessions
TEATT Licenses (BLISS)
Protocol: D365 OData v4 — technically documented
Risk: No unified digital case management — status transitions may have undocumented edge cases
Session focus: Licence status enum validation, status transition business rules, D365 OData filter query testing
Action: Document all undiscovered enum edge cases in the issue register immediately
🟢 LOW RISK · Bi-weekly sessions
Chamber of Commerce (COCI)
Protocol: Oracle OData v4 — digital front-end exists
Risk: Board quorum rules may slow decisions on access configuration
Session focus: OData v4 filter queries, CRIB number format validation, director/shareholder entity structure
Action: Pre-agree access configuration decisions before board meeting cycle to avoid delays
🟢 LOW RISK · Bi-weekly sessions
VROMI Permits (ArcGIS)
Protocol: ArcGIS REST — best As-Is documentation of all registries
Risk: eVROMI planned migration — coordinate timing to avoid integration rework
Session focus: WKT geometry parsing, parcel ID spatial query optimisation, permit status validation
Action: Confirm eVROMI go-live date and whether adapter must support both old and new endpoints during transition
⛔ DEFERRED · No sessions yet
Kadaster (Land Registry)
Status: Formally deferred — no As-Is architecture document
Blocker: Cannot design adapter or support sessions without registry documentation
Action: Kadaster focal point must be engaged in Design Phase Week 1
Target: As-Is documentation received within 4 weeks — D6 sessions begin after
Minimum commitment per registry: 4 hours/week from the registry technical contact, formalised in the Data Sharing Agreement. Without a formalised commitment, registries deprioritise validation sessions and sign-off drags into the Build Phase.
D6 · Support Process Session Format · Issue Register · Escalation Path

Session Format, Issue Register & Escalation

Every D6 session produces structured documentation. No issue stays unassigned. Escalation is automatic after 2 business days of non-response — it is not optional.

SESSION FORMAT — PER REGISTRY
Session Structure (30–60 min, bi-weekly or weekly)▾
Opening (5 min): Review action items from previous session — status update per item.
Sandbox walkthrough (20–30 min): Registry technical contact makes live API calls to rg-dgtp-staging. DGTP team observes, notes unexpected behaviour, and logs issues in real time.
Issue resolution (10–15 min): Review open issues. Assign or close.
Next steps (5 min): Agree action items with named owners and due dates. Confirm next session date.
Issue Register — Severity Definitions▾
Critical: Blocks integration entirely — sandbox returns no valid response or auth fails. Target resolution: 2 business days.
High: Spec inaccuracy or missing field that affects core integration logic. Target: 5 business days.
Medium: Edge case behaviour not documented in the spec. Target: 10 business days.
Low: Documentation clarification or naming preference. Target: next session.
API Call Validation Checklist▾
For each endpoint in the spec pack, the registry technical contact must validate in sandbox:
☐ Authentication flow completes (JWT acquired and accepted)
☐ Happy path: 200 response with correct schema
☐ 404: returns RFC 7807 Problem Details body
☐ 401: returns correct WWW-Authenticate header
☐ 429: returns Retry-After header and X-RateLimit-* headers
☐ Rate limit: 500 req/min enforced at APIM
☐ Cross-registry linkage fields (NSN, CRIB) present and correctly formatted
Sign-Off Confirmation Requirements▾
Written confirmation from the registry technical contact (email or signed document) stating:
1. The OpenAPI 3.0 spec accurately reflects the registry system capabilities
2. All validation checklist items have been completed in sandbox
3. All Critical and High issues are resolved
4. The registry can implement the integration in the Build Phase

Sign-off is required before D6 is closed for that registry and before any live data exchange begins per P-15 Compliance by Design Compliance by Design.
ESCALATION PATH — THREE LEVELS
L1
Registry Focal Point — 2 Business Days
Registry technical contact unresponsive or unable to make progress. Escalate to the registry focal point (management level) within 2 business days. Provide written summary of blocking issue and impact on timeline.
L2
Project Steering Committee — 5 Business Days
Registry issue unresolved after 5 business days from L1 escalation. Formal written notice to SXM Project Steering Committee with: issue description, timeline impact, blocking party, and proposed resolution. PSC must respond within 3 business days.
L3
Mock Mode — Build Phase Does Not Wait
Registry integration blocked for more than 2 weeks. The registry adapter enters mock mode — using the staging mock dataset for the entire Build Phase. Build Phase does not wait for any single registry. Document the mock-mode decision formally and notify Project Sponsor.
Open Question
Who is the single DGTP point of contact per registry for D6 — and is that role formalised?
Each registry needs a named DGTP contact they can reach directly. Without this, registries escalate to project management for technical questions — slowing resolution. Assign and communicate per-registry DGTP contacts at the D5 delivery meeting.

⚙ ARCHITECTURE PRINCIPLES — P-01 to P-15 · Click any row to expand description
P-01
API-First
All platform APIs documented in OpenAPI 3.0 and agreed by Architecture Team BEFORE development begins. GoSXM API Catalogue is the single source of truth.
▾
P-02
Cloud-Native Azure
Exclusive Azure PaaS in European region. No multi-cloud, no on-premise in Phase 1. All deployments via IaC — no manual portal changes in staging or production.
▾
P-03
Separation of Concerns
Seven enforced platform layers. Each layer has one responsibility. No cross-layer shortcutting. Experience Layer cannot query registries directly.
▾
P-04
Loose Coupling
One dedicated, independently deployable adapter per registry. A registry changing its protocol affects only its adapter — zero platform-wide impact.
▾
P-05
Resilience by Default
Outbox Pattern in all adapters guarantees zero data loss. Service Bus zone-redundant with Geo-DR. Circuit breakers prevent cascade failures.
▾
P-06
Open Standards
REST/JSON, OpenAPI 3.0, OAuth2/OIDC, OData where available. Vendor lock-in minimised. All interface contracts machine-readable and publicly documented.
▾
P-07
Security by Design
Security embedded at every layer from Day 1. mTLS between all services. OWASP review is a go-live gate. SAST/CVE scanning in all CI/CD pipelines.
▾
P-08
Identity-First Access
Azure Entra External ID is the sole identity provider. 7 RBAC roles. JWT 15-min TTL. PIM Just-In-Time for all privileged access. No standing admin privileges.
▾
P-09
Privacy by Design
Data minimisation and consent management built into every cross-registry call. Citizens can view their data access history. 90-day PII audit log in Log Analytics.
▾
P-10
Authoritative Source
Each data domain has exactly one authoritative registry. Canonical Data Store is a read-model — it never becomes the source of truth. Registries own their data.
▾
P-11
Once-Only Principle
Citizens submit data once. The platform reuses it across services via the Canonical Data Store with a freshness SLA of less than 15 minutes. No re-entry of known data.
▾
P-12
Data Sovereignty
All citizen data stays within Azure European region at all times. Geo-DR secondary must also be European. No third-party SaaS without a signed EU data processing agreement.
▾
P-13
Observability First
Structured logging, distributed tracing, and metrics from Day 1 in every component. Azure Monitor + Application Insights. Alerts on DLQ first-message within 2 minutes.
▾
P-14
Infrastructure as Code
All Azure resources provisioned via Bicep/Terraform. No manual portal changes in staging or production. IaC enforces region and security constraints in every template.
▾
P-15
Compliance by Design
ID-Exchange EA Ordinance approval is a hard go-live gate. Privacy Ordinance compliance built into data flows. Audit logs satisfy statutory reporting requirements.
▾
