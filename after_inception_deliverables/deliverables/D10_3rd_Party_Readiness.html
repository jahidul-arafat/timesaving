3rd-Party & SaaS Readiness

Every external service the DGTP platform depends on must be assessed before the Build Phase begins. EU data processing agreements, GDPR compliance evidence, SLA requirements, fallback strategies, and vendor lock-in risk rated for each dependency. No external service enters production without a signed DPA.

P-12 Data Sovereignty
Data Sovereignty — EU DPAs
Every 3rd-party or SaaS service that processes citizen data must have a signed EU Data Processing Agreement. The DPA must confirm: data stays within the EU, subprocessors are disclosed and EU-compliant, breach notification within 72 hours, and data deletion on contract termination. No DPA = no production use.
DPA MANDATORY
P-09 Privacy by Design
Privacy by Design — GDPR Evidence
Each vendor must provide a completed GDPR Article 28 processor assessment, a record of processing activities, and a list of all sub-processors. Vendors must demonstrate privacy-by-design in their platform — not just in contractual commitments. Assessed by the DGTP Privacy Officer before production onboarding.
ART. 28 ASSESSMENT
P-15 Compliance by Design
Compliance by Design — Risk Register
All 3rd-party dependencies are entered in the platform risk register with: vendor lock-in risk rating, SLA vs platform SLA compatibility, fallback/exit strategy, GDPR compliance evidence, security certification (ISO 27001 or equivalent), and annual review date. No unregistered 3rd-party dependencies in any environment.

High Vendor Lock-In Risk — Accept or Mitigate?▾

GitHub EU data residency for repositories must be explicitly configured — it is not the default. Action: when creating the DGTP GitHub organisation, select EU data residency in the organisation settings. GitHub Actions runners must be configured to use EU-region hosted runners only. Verify configuration in D9 environment setup.
GovPass SLA Compatibility — TBC▾
GovPass / SXM eID is a government-operated citizen identity service. Its SLA is not yet confirmed for DGTP integration use. If GovPass has a lower SLA than the DGTP platform (99%), citizens will be unable to authenticate when GovPass is down. A fallback authentication path (e.g. email OTP) must be defined in D1 (Security Architecture) if GovPass SLA cannot be guaranteed at 99%.
Esri ArcGIS — Best Vendor Posture▾
Esri ArcGIS has the strongest third-party vendor posture: EU-hosted instance confirmed, GDPR DPA already signed by VROMI, ISO 27001 certified, lowest lock-in risk (standard REST interface), and 99.5% SLA. The VROMI ArcGIS adapter is the lowest-risk integration in the entire platform. Use VROMI as the reference implementation for other adapters.
Open Question — P-12 Data Sovereignty
Does the EA Ordinance impose additional data residency requirements beyond EU region for Sint Maarten citizen data?
If the EA Ordinance requires data to be hosted within the Kingdom of the Netherlands specifically (not just EU), all Azure resource group locations may need to change. Architecture Team must confirm this interpretation with the SXM Legal Counsel before finalising the resource group location policy in D9.

D10 · P-14 Infrastructure as Code · P-15 Compliance by Design Vendor Exit Strategies & Fallback Patterns

Exit Strategies & Fallback Patterns

Every high lock-in vendor must have a documented exit strategy and a fallback pattern for outage. Government platforms must be able to function when a vendor is unavailable — even temporarily.

EXIT STRATEGIES — HIGH LOCK-IN VENDORS
Microsoft Azure — Strategic Accept + Portability▾
Exit strategy: Lock-in risk accepted as a strategic decision per P-02 Cloud-Native Azure Cloud-Native Azure. Mitigation: IaC templates written in standard Bicep/ARM — migrating requires template rewrites, not application rewrites. All application code is .NET 8 targeting standard OpenTelemetry, standard SQL, and standard AMQP (Service Bus) — all have multi-cloud equivalents.

Outage fallback: Azure Geo-DR configured for all data stores (West Europe primary, North Europe secondary). RTO 4 hours. RPO 1 hour. Service Bus Geo-DR enables message replay. Canonical Data Store can operate in read-only mode during primary region outage — citizens can view existing data but not trigger new registry syncs.
Azure Entra External ID — Fallback Auth▾
Exit strategy: Entra is the sole identity provider per P-08 Identity-First Access. Migration to another IDP (Keycloak, Okta) would require: all app registrations recreated, token validation policies updated in APIM, GovPass federation reconfigured. High effort but well-documented migration path.

Outage fallback: If Entra is unavailable, new logins fail but existing JWT tokens (15-minute TTL) remain valid for their remaining lifetime. Staff with active sessions can continue for up to 15 minutes. Emergency read-only mode: APIM can be configured to accept a pre-issued emergency token with 24-hour TTL for critical government staff — requires Architecture Team + Integration Admin dual approval.
GovPass / SXM eID — Email OTP Fallback▾
Exit strategy: Government-to-government federation — no vendor exit strategy needed. If eID changes protocol, the Entra OIDC federation configuration is updated.

Outage fallback: If GovPass is unavailable, citizens cannot authenticate. Fallback: email OTP (Azure Communication Services) for citizens with a verified government email. Citizens without email fallback see a service unavailability message with an estimated restoration time and a phone number for the government contact centre.
Registry Vendors — Outbox Pattern Provides Resilience▾
Exit strategy: Each registry backend vendor (Oracle/COCI, Esri/VROMI, Microsoft D365/TEATT) is isolated to its own adapter per P-04 Loose Coupling Loose Coupling. Replacing a vendor means rewriting one adapter only — no platform-wide impact.

Outage fallback: When a registry is unavailable, the adapter circuit breaker opens and the Canonical Data Store serves the last-known data. Citizens see a freshness indicator showing when data was last updated. New registry syncs queue in the Outbox and deliver automatically when the registry recovers — zero data loss per P-05 Resilience by Default.

⚙ ARCHITECTURE PRINCIPLES — P-01 to P-15 · Click any row to expand description
P-01
API-First
All platform APIs documented in OpenAPI 3.0 and agreed by Architecture Team BEFORE development begins. GoSXM API Catalogue is the single source of truth.
▾
P-02
Cloud-Native Azure
Exclusive Azure PaaS in European region. No multi-cloud, no on-premise in Phase 1. All deployments via IaC — no manual portal changes in staging or production.
▾
P-03
Separation of Concerns
Seven enforced platform layers. Each layer has one responsibility. No cross-layer shortcutting. Experience Layer cannot query registries directly.
▾
P-04
Loose Coupling
One dedicated, independently deployable adapter per registry. A registry changing its protocol affects only its adapter — zero platform-wide impact.
▾
P-05
Resilience by Default
Outbox Pattern in all adapters guarantees zero data loss. Service Bus zone-redundant with Geo-DR. Circuit breakers prevent cascade failures.
▾
P-06
Open Standards
REST/JSON, OpenAPI 3.0, OAuth2/OIDC, OData where available. Vendor lock-in minimised. All interface contracts machine-readable and publicly documented.
▾
P-07
Security by Design
Security embedded at every layer from Day 1. mTLS between all services. OWASP review is a go-live gate. SAST/CVE scanning in all CI/CD pipelines.
▾
P-08
Identity-First Access
Azure Entra External ID is the sole identity provider. 7 RBAC roles. JWT 15-min TTL. PIM Just-In-Time for all privileged access. No standing admin privileges.
▾
P-09
Privacy by Design
Data minimisation and consent management built into every cross-registry call. Citizens can view their data access history. 90-day PII audit log in Log Analytics.
▾
P-10
Authoritative Source
Each data domain has exactly one authoritative registry. Canonical Data Store is a read-model — it never becomes the source of truth. Registries own their data.
▾
P-11
Once-Only Principle
Citizens submit data once. The platform reuses it across services via the Canonical Data Store with a freshness SLA of less than 15 minutes. No re-entry of known data.
▾
P-12
Data Sovereignty
All citizen data stays within Azure European region at all times. Geo-DR secondary must also be European. No third-party SaaS without a signed EU data processing agreement.
▾
P-13
Observability First
Structured logging, distributed tracing, and metrics from Day 1 in every component. Azure Monitor + Application Insights. Alerts on DLQ first-message within 2 minutes.
▾
P-14
Infrastructure as Code
All Azure resources provisioned via Bicep/Terraform. No manual portal changes in staging or production. IaC enforces region and security constraints in every template.
▾
P-15
Compliance by Design
ID-Exchange EA Ordinance approval is a hard go-live gate. Privacy Ordinance compliance built into data flows. Audit logs satisfy statutory reporting requirements.
▾
