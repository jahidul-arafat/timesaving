Data Schema & Outbox Pattern Implementation

The Canonical Data Store schema defines the unified citizen data model. The Outbox Pattern implementation guarantees zero data loss per P-05 Resilience by Default Resilience by Default. Hover each entity for field definitions.

CANONICAL DATA SCHEMA — ENTITY RELATIONSHIPS · HOVER FOR FIELD DEFINITIONS
DATA DESIGN DECISIONS — P-10 Authoritative Source · P-11 Once-Only Principle · P-12 Data Sovereignty
NSN as the Universal Citizen Key (P-10 Authoritative Source)▾
The National Sofi Nummer (NSN) is the primary citizen identifier across all six registries. Every cross-registry entity that represents a person includes an NSN foreign key back to the Civil Registry Person entity. The Canonical Data Store uses NSN as the join key for all once-only lookups. The NSN field is validated against the pattern ^[0-9]{9}$ (historical 8-digit NSNs handled via migration mapping table).
Freshness tracking per entity (P-11 Once-Only Principle)▾
Every canonical entity includes a freshnessStatus field: FRESH (updated within SLA), STALE (SLA exceeded but data present), or UNKNOWN (never synced). The <15 minute freshness SLA is monitored per entity type via Azure Monitor. If any entity type's average lag exceeds 12 minutes, an alert fires before the SLA is breached. Consumers can query freshnessStatus to decide whether to use cached data or trigger a refresh.
Outbox Pattern implementation (P-05 Resilience by Default)▾
Each adapter has its own OutboxEvent table in its adapter database. Adapter writes entity data + OutboxEvent record in a single DB transaction (atomic). Background publisher polls PENDING records every 30 seconds. On successful Service Bus publish: status → PUBLISHED, publishedAt set. On failure: attempts incremented. After 3 attempts: status → FAILED, DLQ alert fires within 2 minutes.
Data residency enforcement (P-12 Data Sovereignty)▾
All Cosmos DB and Azure SQL instances are provisioned in West Europe (primary) with failover to North Europe (secondary) — both within the EU. IaC templates enforce this via the allowedLocations Azure Policy. Geo-DR for Service Bus and Cosmos DB is configured to North Europe only. Any attempt to create a resource outside EU regions is blocked by Azure Policy at the subscription level.
LLD Decision — Cosmos DB vs Azure SQL
Which entities go in Cosmos DB (document) vs Azure SQL (relational)?
Recommendation: Person, Business, Licence, Permit in Azure SQL (strong relational joins needed for once-only lookups). OutboxEvent tables in adapter-specific SQL. Geospatial parcel data (WKT geometry) in Cosmos DB with geospatial index. Architecture Team must ratify this split in D8 Section 4.2 before Build Phase.

⚙ ARCHITECTURE PRINCIPLES — P-01 to P-15 · Click any row to expand description
P-01
API-First
All platform APIs documented in OpenAPI 3.0 and agreed by Architecture Team BEFORE development begins. GoSXM API Catalogue is the single source of truth.
▾
P-02
Cloud-Native Azure
Exclusive Azure PaaS in European region. No multi-cloud, no on-premise in Phase 1. All deployments via IaC — no manual portal changes in staging or production.
▾
P-03
Separation of Concerns
Seven enforced platform layers. Each layer has one responsibility. No cross-layer shortcutting. Experience Layer cannot query registries directly.
▾
P-04
Loose Coupling
One dedicated, independently deployable adapter per registry. A registry changing its protocol affects only its adapter — zero platform-wide impact.
▾
P-05
Resilience by Default
Outbox Pattern in all adapters guarantees zero data loss. Service Bus zone-redundant with Geo-DR. Circuit breakers prevent cascade failures.
▾
P-06
Open Standards
REST/JSON, OpenAPI 3.0, OAuth2/OIDC, OData where available. Vendor lock-in minimised. All interface contracts machine-readable and publicly documented.
▾
P-07
Security by Design
Security embedded at every layer from Day 1. mTLS between all services. OWASP review is a go-live gate. SAST/CVE scanning in all CI/CD pipelines.
▾
P-08
Identity-First Access
Azure Entra External ID is the sole identity provider. 7 RBAC roles. JWT 15-min TTL. PIM Just-In-Time for all privileged access. No standing admin privileges.
▾
P-09
Privacy by Design
Data minimisation and consent management built into every cross-registry call. Citizens can view their data access history. 90-day PII audit log in Log Analytics.
▾
P-10
Authoritative Source
Each data domain has exactly one authoritative registry. Canonical Data Store is a read-model — it never becomes the source of truth. Registries own their data.
▾
P-11
Once-Only Principle
Citizens submit data once. The platform reuses it across services via the Canonical Data Store with a freshness SLA of less than 15 minutes. No re-entry of known data.
▾
P-12
Data Sovereignty
All citizen data stays within Azure European region at all times. Geo-DR secondary must also be European. No third-party SaaS without a signed EU data processing agreement.
▾
P-13
Observability First
Structured logging, distributed tracing, and metrics from Day 1 in every component. Azure Monitor + Application Insights. Alerts on DLQ first-message within 2 minutes.
▾
P-14
Infrastructure as Code
All Azure resources provisioned via Bicep/Terraform. No manual portal changes in staging or production. IaC enforces region and security constraints in every template.
▾
P-15
Compliance by Design
ID-Exchange EA Ordinance approval is a hard go-live gate. Privacy Ordinance compliance built into data flows. Audit logs satisfy statutory reporting requirements.
▾
