WHAT EACH SPEC PACK CONTAINS
① OpenAPI 3.0 Spec Bundle▾
The approved OpenAPI 3.0 spec for that registry. All endpoints, request/response schemas, error codes (RFC 7807 format), authentication patterns, and rate limit documentation. Swagger UI rendering link on Azure APIM developer portal for interactive exploration.
② Authentication Setup Guide▾
Step-by-step OAuth2 client registration instructions for the registry technical team. Covers: Entra External ID app registration, client credentials setup (M2M), sandbox token acquisition, JWT validation, and RBAC role assignment for the registry's integration account.
③ Sandbox Connection Instructions▾
Dedicated sandbox credentials for the registry technical contact. rg-dgtp-staging base URL, mock data schema per registry, how to make test calls via APIM, expected response shapes, and error scenario testing guide. Sandbox must be fully provisioned before this section can be completed.
④ Data Sharing Agreement Template▾
Legal template for the registry's legal/management team to review and sign. Covers: data scope (what fields are shared), purpose limitation, consent requirements, retention periods, access controls, P-12 Data Sovereignty Data Sovereignty clause (EU region only), breach notification obligations, and review frequency. Must be signed before any live data exchange begins.
Timing Risk
Can data sharing agreements be negotiated and signed within the 3-week D5 delivery window?
DSA negotiations typically take 4–8 weeks. If started only after D4 completes, they will delay registry implementation. Response: begin DSA drafting in Design Phase Week 1 — in parallel with D4, not after it.

Sandbox Environment & Spec Pack Governance

rg-dgtp-staging is the integration test environment. Registry technical contacts use it to validate the spec pack before sign-off. Governance defines what happens when a registry refuses or requests spec changes.

Sandbox Must Be Ready Before D5 Delivery Begins▾
The sandbox environment (rg-dgtp-staging) must be fully provisioned and validated — with mock data per registry and working APIM mock policies — before any spec pack can be delivered. Registry technical contacts must be able to make real API calls and see real (mock) responses. Sandbox provisioning is a parallel track to D4, not a dependency that starts after D4 completes.
Mock Data Requirements Per Registry▾
Each registry mock dataset must contain: at least 10 representative records covering all status types and edge cases, cross-registry linkage records (NSN → CRIB → licence chains), error scenario triggers (invalid IDs, expired tokens, rate limit responses), and WKT geometry samples for VROMI. No real citizen data permitted in staging — all mock data must be synthetically generated.
Sandbox Credentials — Per Registry Contact▾
Each registry technical contact receives dedicated OAuth2 client credentials scoped to their registry's sandbox endpoints only. Credentials expire after 90 days and must be renewed. Credentials are delivered via a secure channel — never by email in plain text. The DGTP Integration Admin manages credential issuance and rotation in Azure Entra External ID per P-08 Identity-First Access Identity-First Access.
If a Registry Refuses to Sign the DSA▾
No live data exchange is permitted. The registry adapter operates in mock mode only — using the staging mock dataset. Escalate to Project Steering Committee with a formal written notice immediately. No exceptions to this rule: a refusal to sign means no live integration, regardless of timeline pressure.

Open Question
What is the minimum time commitment per week required from each registry technical contact to meet the D5 validation timeline?
D5 requires active participation from registry staff who also have full-time operational duties. A minimum 4 hours/week commitment should be formalised in the DSA to protect the timeline. Without it, validation drags into the Build Phase.

⚙ ARCHITECTURE PRINCIPLES — P-01 to P-15 · Click any row to expand description
P-01
API-First
All platform APIs documented in OpenAPI 3.0 and agreed by Architecture Team BEFORE development begins. GoSXM API Catalogue is the single source of truth.
▾
P-02
Cloud-Native Azure
Exclusive Azure PaaS in European region. No multi-cloud, no on-premise in Phase 1. All deployments via IaC — no manual portal changes in staging or production.
▾
P-03
Separation of Concerns
Seven enforced platform layers. Each layer has one responsibility. No cross-layer shortcutting. Experience Layer cannot query registries directly.
▾
P-04
Loose Coupling
One dedicated, independently deployable adapter per registry. A registry changing its protocol affects only its adapter — zero platform-wide impact.
▾
P-05
Resilience by Default
Outbox Pattern in all adapters guarantees zero data loss. Service Bus zone-redundant with Geo-DR. Circuit breakers prevent cascade failures.
▾
P-06
Open Standards
REST/JSON, OpenAPI 3.0, OAuth2/OIDC, OData where available. Vendor lock-in minimised. All interface contracts machine-readable and publicly documented.
▾
P-07
Security by Design
Security embedded at every layer from Day 1. mTLS between all services. OWASP review is a go-live gate. SAST/CVE scanning in all CI/CD pipelines.
▾
P-08
Identity-First Access
Azure Entra External ID is the sole identity provider. 7 RBAC roles. JWT 15-min TTL. PIM Just-In-Time for all privileged access. No standing admin privileges.
▾
P-09
Privacy by Design
Data minimisation and consent management built into every cross-registry call. Citizens can view their data access history. 90-day PII audit log in Log Analytics.
▾
P-10
Authoritative Source
Each data domain has exactly one authoritative registry. Canonical Data Store is a read-model — it never becomes the source of truth. Registries own their data.
▾
P-11
Once-Only Principle
Citizens submit data once. The platform reuses it across services via the Canonical Data Store with a freshness SLA of less than 15 minutes. No re-entry of known data.
▾
P-12
Data Sovereignty
All citizen data stays within Azure European region at all times. Geo-DR secondary must also be European. No third-party SaaS without a signed EU data processing agreement.
▾
P-13
Observability First
Structured logging, distributed tracing, and metrics from Day 1 in every component. Azure Monitor + Application Insights. Alerts on DLQ first-message within 2 minutes.
▾
P-14
Infrastructure as Code
All Azure resources provisioned via Bicep/Terraform. No manual portal changes in staging or production. IaC enforces region and security constraints in every template.
▾
P-15
Compliance by Design
ID-Exchange EA Ordinance approval is a hard go-live gate. Privacy Ordinance compliance built into data flows. Audit logs satisfy statutory reporting requirements.
▾
