Component 01 — APIM Gateway

APIM 8-Stage Inbound Policy Pipeline

Marcus taps "Get Tax Clearance" in GoSXM Portal
"GET /v1/tax/clearance?cribNumber=123456789" — bearer token in header
This single HTTP request passes through 8 sequential policy stages before it reaches the SIAH adapter Function App. Each stage can terminate the request with an error code. Only a request passing all 8 reaches the registry. This is where rate limits, IP filtering, JWT validation, and backend auth all live — in declarative XML policies, version-controlled in /infra/apim/policies/.

Audience Check · Data & Events
Data flows, dead letters, and dangerous assumptions
01
An adapter crashes halfway through writing a COCI update. Does the message get lost?
tap to reveal →
No.
The Outbox pattern ensures zero data loss. The adapter writes to both the CMF cache and dbo.OutboxMessages in a single database transaction. If the adapter crashes before the Service Bus publish, the row stays PENDING. The timer-triggered relay picks it up on the next run. No message is ever lost — only delayed.
02
A DLQ message is 4 days old and nobody noticed. What just happened in the ops process?
tap to reveal →
P2 breach
DLQ messages older than 3 days trigger a P2 incident automatically via Azure Monitor alert. A 4-day-old message means either the alert fired and was ignored — or the alert rule was not configured. Both are ops failures. The DLQ monitor timer fires every 15 minutes. This is the canary for subscription health.
03
The gosxm-notification-sub is missing from the production Service Bus. Is that a bug?
tap to reveal →
Correct.
The portal push-notification subscription is Phase 2 only — intentionally excluded from Phase 1 scope. It must be confirmed absent in Terraform configuration for Phase 1. Adding it accidentally would create a subscription with no consumer, which would fill the DLQ and trigger false P2 alerts. Confirm with the portal team before Phase 2 sprint planning.
Component 06 — Identity

Identity Layer — Entra External ID & eID Federation

Q2 answer — why the architecture is identity-provider agnostic
"eID provider changes managed through federation config updates only — no application redesign, no adapter changes, no portal code changes."
Azure Entra External ID acts as the federation and identity brokering layer. It federates with the national eID via OIDC, issues JWT tokens with scopes, and manages the session. If Sint Maarten's eID provider changes, only the Entra federation config changes. Everything downstream is decoupled.

Q2 — eID & Entra: Provider-Agnostic

Entra federates with the national eID via standard OIDC. If Sint Maarten changes eID provider: update federation config in Entra only. Zero changes to DGTP services, adapters, or portal code.

01
A service account tries to pull Marcus's tax clearance from SIAH. What happens — and why?
tap to reveal →
Hard block.
LLD §16.2 constraint: the SIAH adapter checks for a citizen OIDC token on every call. A service-account bearer token does not carry a citizen subject claim — the adapter rejects it before any OData query is made. This cannot be overridden by APIM scope configuration. Tax data goes only to the citizen who owns it, in their own authenticated session.
02
Sint Maarten switches to a new national eID provider. How many lines of DGTP code change?
tap to reveal →
Zero.
The architecture is identity-provider agnostic. Azure Entra External ID brokers between the national eID and all DGTP downstream services. A provider change means updating the OIDC federation configuration in Entra — a config change, not a code change. No adapter, no pipeline stage, no portal component knows which eID provider is behind Entra.
03
Why is SOAR automated response deliberately off in Phase 1 — even though the tech works?
tap to reveal →
Alert quality.
Activating automated response before alert thresholds are tuned and validated creates false-positive automated actions — blocking legitimate traffic, isolating services unnecessarily, eroding trust in the system. Phase 1 is the tuning period. Orangebd triages every alert manually. Once alert quality is confirmed (Phase 2), SOAR automation is safe to activate. Automation is a multiplier — of good ops and bad ops.
