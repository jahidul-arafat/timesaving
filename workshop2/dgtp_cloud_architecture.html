Three Adaptable Plans

The XLS PROD v2 aggressive stack is over-engineered for 50,000 citizens at launch. These three plans let DGTP start lean, validate demand, and scale only what the data justifies.

A
Minimal Cloud
Core platform · lean stack · go-live ready

Everything needed to run DGTP end-to-end for 50,000 citizens — and nothing more. Mirrors the optimised DEV environment pattern scaled for production use. Single instances, no geo-redundancy, Standard service tiers, Basic firewall. Sentinel excluded. AI Search on free tier.

✓All 6 registry adapters live
✓APIM + Entra + Service Bus Standard
✓SQL MI GP 4 vCore + Cosmos Serverless
✓Front Door Standard + Firewall Basic
✗No DDoS Standard · No Sentinel · No HA
✗No zone-redundancy · No geo-DR
Best for: Phase 1 Go-Live · Validate demand · First 6–12 months

Upgrade path: each component swappable independently — add DDoS Standard first, then HA ASE, then Sentinel SOAR

B
Standard Cloud
Balanced · prod-pattern fidelity · right-sized

Matches the XLS optimised STG configuration scaled for production. Basic HA (2-instance ASE), Front Door Premium for WAF, Sentinel active, Service Bus Standard with quarterly Premium validation windows. SQL MI GP 8 vCore. Cosmos autoscale. Firewall Premium for IDPS.

✓Front Door Premium + WAF v2
✓2-instance ASE (basic HA per tier)
✓Sentinel + Defender P2 active
✓Firewall Premium · VPN Gateway

Full Cloud
PROD v2 aggressive · full enterprise · all NFRs met

The XLS PROD v2 aggressive configuration in full. 3-AZ zone-redundant ASE (3 frontend + 4 backend), DDoS Standard, SQL MI GP 16 vCore with 1yr RI, Cosmos autoscale 50k RU/s, Service Bus Premium 1MU with geo-DR, AI Search S2 ×2, full Sentinel + Purview 8CU. All locked items active.

A
Plan A · Minimal Cloud

Start here. Get citizens online. Spend only what you need.

Non-technical analogy
Think of Plan A as renting a reliable car — not buying a fleet. It gets you everywhere Sint Maarten's 50,000 citizens need to go. If demand grows and you need more, you upgrade. You don't buy the fleet before you know the routes.
What Plan A includes
🚪
The front door
Azure Front Door Standard + App Gateway
Citizens reach the GoSXM portal through a global content delivery network with basic security rules. Think of it as a secure reception desk — it checks everyone at the door, handles the queue, and routes them to the right service. The Standard tier includes protection against the most common web attacks.
⚙️
The engine room
1 server per tier — all 6 registries connected
One server handles the front-end, one handles the back-end processing. No backup server — if this server has a problem, there is a brief outage. For 50,000 citizens at launch, a single reliable server is the right call. All six government registries (Civil, Commerce, Tax, TEATT, Permits, Land) are connected and live.
🗄️
The data safe
SQL Database + Smart Cache + Audit Log
Citizen data is stored securely in a SQL database with a 4-core processor. A smart cache means most portal requests are answered in milliseconds — not by calling the registry every time. Every action is permanently logged for 7 years (legal requirement), and citizen identities are stored as scrambled codes — never as plain names.
🔒
Security baseline
Firewall + Identity + Secrets Vault
A network firewall blocks all uninvited traffic. Citizens log in with the national eID via a secure identity service (Entra). All system passwords and API keys are stored in a locked vault — no credentials are ever hardcoded or visible. Security event logs are monitored via Azure Defender.
What Plan A does NOT include — and why that's fine
🛡️
No DDoS Standard protection
Front Door edge protection is sufficient at launch
DDoS Standard costs $2,944/month and protects the entire network perimeter. For a 50,000-citizen government portal at launch, the Azure Front Door edge protection (included in Standard tier) handles realistic attack scenarios. DDoS Standard becomes important when Sint Maarten's digital services are a high-value target — not at Go-Live.
📊
No Sentinel (security operations centre)
Security investigations via Azure Defender alerts
Microsoft Sentinel is a full security intelligence platform — powerful, but adds significant cost and requires a trained security team to interpret its findings. Without a dedicated SOC, activating Sentinel creates noise, not insight. Plan A uses Azure Defender alerts and log queries instead — the same data, simpler tooling.
⚖️
No high-availability failover
Single server — single point of failure
If the single server has a problem, there is a brief service interruption. For a portal serving 50,000 citizens with moderate usage patterns, this risk is acceptable at launch. A brief outage during the first year — while you gather real usage data — is less costly than over-spending on infrastructure that may never be needed.
When to upgrade from Plan A
📈
Portal traffic grows beyond projections — response times slow, or the server CPU is consistently above 70%
⚖️
Regulator mandates DDoS Standard — this is the single biggest cost jump; only commit when required
🔐
A security incident occurs that requires Sentinel's advanced threat hunting capability
🏛️
GoSXM establishes a dedicated SOC — then Sentinel becomes valuable, not noise
Plan A includes: All 6 registry adapters APIM 8-stage pipeline eID login via Entra 7yr audit log CMF data cache Service Bus events Terraform IaC Secrets vault DDoS Standard Sentinel HA failover Zone-redundancy Geo-DR
B
Plan B · Standard Cloud

Proven demand. Production-ready. Balanced investment.

Non-technical analogy
Plan B is like upgrading from one checkout lane to two in a supermarket — you can handle twice the queue without turning anyone away. You've proven the shop is popular. Now you invest in the second lane, better security cameras, and a manager watching the floor.
What Plan B adds over Plan A
🔁
Basic high-availability
2 servers per tier — if one fails, the other takes over
Plan B runs two servers instead of one at each tier. If one has a problem, the other continues without interruption. This is the minimum for a government service that citizens are relying on daily. The switchover is automatic — citizens notice nothing.
🛡️
Upgraded security inspection
Firewall Premium — reads inside encrypted traffic
Plan B upgrades to a Premium firewall with IDPS — it can inspect encrypted (HTTPS) traffic for threats, not just the outside of the packet. Think of it as upgrading from a metal detector at the door to an X-ray machine — it sees what's inside. Required to validate security rules before deploying to the full production environment.
📡
Security monitoring activated
Microsoft Sentinel — threat detection and alerting
Plan B turns on Microsoft Sentinel — a security intelligence platform that monitors all activity and flags suspicious patterns. It's the difference between a security camera that records and one that alerts the guard when it sees something unusual. At advisory mode initially — alerts are reviewed by the Orangebd team before automated responses are activated.
🔗
Premium WAF + secure origin connections
Front Door Premium — encrypted private connections to servers
Plan B upgrades the front door to Premium tier, which allows the CDN to connect to backend servers via private encrypted links (not public internet). Citizen data never travels over the public internet between the front door and the servers. Also enables advanced WAF rules for sophisticated attack patterns.
What stays the same as Plan A
✅
Unchanged components
All 6 registry connections work identically. The citizen login process (eID + Entra) is unchanged. The 7-year audit log and SHA-256 identity scrambling are unchanged. The CMF data cache, Service Bus message system, and Terraform infrastructure-as-code pipelines are identical. Only the scale and security tier change — not the architecture.
What Plan B still does NOT include
⚡
No DDoS Standard — quarterly drill only
DDoS Standard is still not deployed continuously. Instead, it is activated for a 4-hour window each quarter to validate the configuration — then switched off to avoid the $2,944/month cost. This is a conscious decision, not an oversight. The regulator must mandate continuous DDoS Standard before it becomes a fixed cost.
🌍
No geographic disaster recovery
If the entire Azure data centre region has an outage, Plan B has no automatic failover to a second region. For Sint Maarten's usage patterns, a regional Azure outage is extremely rare and brief. The Cosmos DB and SQL databases have local backups that allow recovery within hours — not days.
🏗️
No 3-zone spread
Azure data centres are divided into availability zones (physically separate buildings with separate power and cooling). Plan B runs 2 servers but does not guarantee they are in separate zones. Plan C guarantees 3 separate zones — Plan B's 2 servers may share a zone.
When to upgrade from Plan B to Plan C
📊
Load data from Plan A shows sustained high CPU, database queries, or cache misses over 3+ months
⚖️
Formal RTO/RPO SLA commitment to the Government requires <5 minute recovery time objective
🔐
PCI-DSS scope confirmed — triggers Key Vault Premium HSM + ADR-005 update + QSA sign-off
Plan B adds: 2 servers per tier (basic HA) Front Door Premium + WAF Firewall Premium + IDPS Sentinel monitoring Defender P2 all plans VPN Gateway (on-prem link) Bastion Standard DDoS Standard (quarterly drill) 3-AZ zone spread Geo-DR
C
Plan C · Full Cloud

The full enterprise stack — when the data says you need it.

Non-technical analogy
Plan C is like opening three checkout lanes, each in a separate building, with a full security operations team, an armoured van, and a backup warehouse in another city. It is what a national bank or a high-traffic e-commerce platform would deploy. Sint Maarten may need this — but only once the traffic data justifies it.
⚠️
Important for decision-makers: Plan C requires two formal Architecture Decision Records (ADR-011 and ADR-005 update) and a QSA (security auditor) sign-off before the SQL and Key Vault changes can be committed. These are not optional — they are compliance obligations. Plan C is not a simple upgrade; it is a formal commitment.
What Plan C adds over Plan B
🏛️
3-zone availability — three physically separate buildings
7 servers across 3 Azure availability zones
Plan C runs 3 frontend servers and 4 backend servers, each in a physically separate building within the same Azure region — separate power, separate cooling, separate network. If one entire building has a problem, the other two continue without any citizen noticing. This is the foundation of the 99.9%+ SLA commitment.
🛡️
DDoS Standard — full network perimeter protection
Always-on attack absorption for the entire network
This is the most expensive single addition ($2,944/month). DDoS Standard protects the entire Azure VNet from volumetric attacks — not just the front door. A DDoS attack at scale could flood Sint Maarten's network perimeter, not just the website. Required once the platform is a high-value regulatory target.
🔑
Hardware security module (HSM) for encryption keys
Key Vault Premium — government-grade key protection
Plan C uses a hardware security module (FIPS 140-2 Level 2) to protect encryption keys. This is a physical chip that generates and stores keys — they can never be exported, even by Microsoft engineers. Required if PCI-DSS scope is confirmed for Sint Maarten's payment flows. Needs QSA sign-off.
🌐
Service Bus Premium — dedicated messaging lane
Dedicated message queue — no shared infrastructure
Plans A and B share messaging infrastructure with other Azure customers. Plan C gets a dedicated lane — guaranteed throughput, no noisy neighbours. Also enables geographic disaster recovery for the message queue: if the primary region fails, messages fail over to a backup queue automatically within minutes.
Formal commitments Plan C requires
📋
ADR-011 — database architecture change
SQL Managed Instance General Purpose · load test required
The XLS baseline used SQL Database Business Critical. Plan C uses SQL Managed Instance General Purpose — a different product with significantly lower cost but different performance characteristics. It handles Sint Maarten's expected workload comfortably, but this must be validated with a load test before the Reserved Instance (annual cost commitment) is signed. ADR-011 documents this decision formally.
📋
ADR-005 update — key management change
Managed HSM removed · Key Vault Premium replaces it · QSA review
The XLS baseline included a dedicated Managed HSM ($3,320/month). Plan C replaces it with Key Vault Premium's HSM-backed keys — same security standard, far lower cost. This change requires a formal update to ADR-005 and a sign-off from the QSA (Qualified Security Assessor) if PCI-DSS compliance is in scope.
Items locked in Plan C — cannot be removed
🔒 DDoS Standard
Regulatory requirement. Cannot be removed without regulator sign-off.
🔒 Front Door Premium
Private backend connections required for citizen data SLA.
🔒 ASE HA footprint
Once Reserved Instance committed, 3-instance minimum is contractual.
🔒 Storage GZRS + WORM
7-year immutable storage. Legal compliance. Cannot downgrade.

Locked Items

Components That Cannot Be Reduced Further

Seven components are locked in Plan C — and must remain locked regardless of optimisation pressure. Any attempt to remove or downgrade these requires a formal ADR and in most cases regulatory sign-off.

🔒 DDoS Protection Standard

Regulatory requirement for production public-facing VNet. Front Door edge DDoS protection only is insufficient when the VNet itself is the regulatory boundary. Cannot be removed without regulator sign-off.

Available in: Plan C only · Not in Plan A or B (quarterly drill only)
🔒 Azure Firewall Premium

IDPS (Intrusion Detection and Prevention System) + TLS deep-packet inspection are Premium-only. Required to validate IDPS rules before prod rollout (Plan B+) and locked in Plan C. Plan A uses Basic for topology parity without IDPS.

Available in: Plan B and C · Plan A uses Firewall Basic
🔒 Azure Front Door Premium

WAF Premium (OWASP + custom rules) + private link origins for backend pools. Standard tier cannot connect to private endpoints — citizen data flows through public origin. Required for prod SLA. Plan A uses Standard.

Available in: Plan B and C · Plan A uses Front Door Standard
🔒 Storage GZRS + WORM 7-year

Geo-zone-redundant storage required for citizen data geo-redundancy SLA. 7-year WORM immutability required for audit log compliance (EA Ordinance). Downgrading to LRS/ZRS or removing WORM is a compliance violation.

Available in: Plan C only · Plans A/B use LRS (non-compliance environments)
🔒 ASE v3 HA Footprint (Plan C)

3 frontend + 4 backend I4v2 instances across 3 AZs is the minimum for the 99.9% HA SLA under the 1-year Reserved Instance commitment. Once the RI is committed, reducing instances below 3 per tier breaks the HA contract. Reserved Instances apply savings.

Cannot reduce after RI commitment · Plans A/B have no RI commitment
🔒 Bastion Standard

Native client support + file transfer required for admin operations (RDP/SSH without public IP). Standard SKU is the admin access pattern — no public IPs on any DGTP compute. Plan A uses Bastion Basic (browser only).

Available in: Plan B and C · Plan A uses Bastion Basic
🔒 Defender for Cloud P2

Security baseline per ADR-009. All P2 plans required: Defender for Servers P2 (JIT VM access), Defender for App Service, Defender for Storage, Defender for SQL. Cannot be downgraded without ADR-009 revision and security team sign-off.

Plan A uses Server P1 only · Plans B and C require full P2
⚠ ADR Changes Required for Plan C

ADR-011 (new): SQL DB Business Critical → SQL MI General Purpose. GP delivers 8,000 IOPS @ 16 vCore. Validate with load test before RI commit.

ADR-005 (update): Managed HSM → Key Vault Premium FIPS 140-2 L2. Multi-tenant HSM vs sole-tenant. QSA sign-off required for PCI-DSS attestation.

Also: SB geo-DR RTO change (~30min vs <5min) · Cosmos single-write write RTO · VPN GW zone-redundancy removed

Recommendation

Start with Plan A.
Upgrade on signal — not on schedule.

Sint Maarten has 50,000 citizens. The XLS PROD v2 baseline was designed for scale that hasn't been proven yet. Plan A gives you a live, auditable, secure platform at a fraction of the cost. Every Plan A component has a direct upgrade path to Plan B and Plan C — independently, without topology changes.

Phase 1 Go-Live · Plan A

Deploy Plan A. All 6 adapters live. Outbox + Service Bus Standard. SQL MI GP 4vCore. Cosmos Serverless. Firewall Basic. No Sentinel, no DDoS. Collect 3 months of production data — CPU, IOPS, request rates, Cosmos RU/s.

Post-UAT · Upgrade to Plan B

When UAT passes and citizen volume is confirmed: swap Front Door Standard → Premium, add 2nd ASE instance per tier, activate Sentinel, upgrade Firewall Basic → Premium, activate VPN Gateway. Each upgrade is a Terraform change — no architecture redesign.

Months 18+ · Plan C on Evidence

Commit to Plan C only when load data justifies it: sustained P95 >2s on Plan B, Cosmos RU/s sustained >10k, citizen volume validated >30k active users. Draft ADR-011 + ADR-005, get QSA sign-off, then commit SQL MI RI + DDoS Standard + zone-redundant ASE.
