Threat Model · STRIDE

What we are protecting against

Before examining controls, we define the threats. The platform applies STRIDE-based threat modelling across all five security layers.

🎭
S — Spoofing
Identity impersonation
Threat: A malicious actor presenting a forged or stolen citizen eID token to gain access.

Control: OIDC/JWT signed with Entra External ID. Token signature verified at APIM Gate ①. Short TTL (2h). mTLS mutual auth on all internal service calls.
Entra External IDJWT RS256mTLS
🔧
T — Tampering
Data modification
Threat: A request payload modified in transit between the portal and a registry adapter, corrupting citizen data.

Control: TLS 1.2+ end-to-end. APIM Gate ⑤ validates payload schema before forwarding. HMAC signatures on payment callbacks. Append-only audit log.
TLS 1.2+Schema validationHMAC
🚫
R — Repudiation
Denying actions
Threat: A citizen or government official denying they accessed or modified a registry record.

Control: Every request receives a unique Tracking ID at Gate ⑥. Structured audit log in Azure SQL — append-only, 7-year retention. Citizen IDs stored as HMAC-hashed values only.
Tracking IDAudit log 7yrHMAC hash
🔓
I — Info Disclosure
Unauthorised data access
Threat: Citizen A accessing citizen B's Civil Registry or Taxpayer records.

Control: APIM Gate ② enforces citizen-scoped OAuth claims. No citizen PII stored in Redis. Cosmos TTL auto-expires cached records. CMF records keyed by scrambled citizen ID.
OAuth scopesNo PII in cacheCosmos TTL

D — Denial of Service
Platform unavailability
Threat: A flood attack targeting the citizen portal or APIM gateway, causing service disruption.

Control: Azure DDoS Standard at perimeter. APIM Gate ③ rate-limits per session. Azure Front Door absorbs volumetric attacks. Multi-AZ deployment — no single point of failure.
DDoS StandardRate limitingMulti-AZ

E — Elevation of Privilege
Privilege escalation
Threat: A citizen token being used to access staff-level operations, or a compromised service accessing another service's data.

Control: RBAC enforced in Entra. Least-privilege service identities (Managed Identities). APIM policies prevent scope elevation. SAST scan catches overpermissioned code.

Marcus · Step-by-step walkthrough

Marcus & the threat model

Why does Marcus's registration need a threat model? Because without it, anyone could impersonate him, steal his records, or take down the system while he's mid-registration.

Marcus · Sole Proprietor · Sint Maarten
Someone tries to pretend they are Marcus. Someone tries to change Marcus's data on the way.Someone tries to read another citizen's records.Someone tries to flood the portal and make it crash.
🎭
Someone tries to pretend they are Marcus
A person steals a copy of Marcus's digital login pass (JWT token) and tries to use it to access his government records without Marcus knowing.
🔐 How the platform stops this: every login pass has a hidden digital stamp that cannot be copied or faked (this is called an RS256 cryptographic signature). Even if someone steals the pass, it stops working on its own after 2 hours (called a TTL — Time To Live). The platform checks the stamp on every single request.
🔧
Someone tries to change Marcus's data on the way
A hacker sitting between Marcus's browser and the platform intercepts his registration request (a man-in-the-middle attack) and tries to change his address before it arrives at the server.
🔐 How the platform stops this: the entire connection between Marcus and the platform is scrambled end-to-end so nobody in the middle can read or change it — the same protection used by online banking (called TLS 1.2+ encryption). The platform also checks that every request has the exact shape it expects before doing anything with it (called schema validation at Gate ⑤).
🔓
Someone tries to read another citizen's records
Another person tries to look up Marcus's tax information by guessing or copying his citizen ID number — an unauthorised data access attempt (information disclosure threat in STRIDE).
🔐 How the platform stops this: Marcus's citizen ID is never stored in readable form — it is run through a one-way mathematical scramble so the original number cannot be recovered (called HMAC hashing). Every request is checked to confirm the person asking is the same person the login pass was issued to. A mismatch is rejected immediately (HTTP 403 — "you are not allowed").
💥
Someone tries to flood the portal and make it crash
An automated bot fires thousands of fake requests per second at the portal, trying to slow it down or crash it (called a DDoS attack — Distributed Denial of Service) while real citizens like Marcus are trying to register.
🔐 How the platform stops this: a flood protection service (called Azure DDoS Standard) watches traffic constantly and automatically absorbs large attack floods before they reach anything. On top of that, each login session is limited to a sensible number of requests per minute (called rate limiting at Gate ③), and a filter at the front door blocks suspicious traffic before it even enters the platform (called Azure Front Door).

Inside the platform, different services live in different sealed network zones (called subnets inside the VNet — Virtual Network). The gateway zone only accepts traffic that has come from the front door — nothing else can get in.

The connector fetches Marcus's Civil Registry data through a private tunnel
The Civil Registry connector needs to contact the Key2DataDistributie system. It does this through a private internal address (called a Private Endpoint) — the connection never leaves the platform's sealed network (VNet).
🔌 What this means technically: instead of connecting to the Civil Registry over the public internet, the connector uses a Private Endpoint — a private internal address that only exists inside the platform's network. The Civil Registry's address is resolved through a private name service (called Private DNS) so it never goes through a public server. The firewall only allows outbound connections from the connector zone to the Civil Registry's specific IP address and port.
🗄️
Marcus's fetched data is stored in a vault with no front door to the outside
After fetching Marcus's details, the platform needs to store them briefly (in Cosmos DB — a fast temporary database) while the rest of his registration completes. This store has no public internet address — it can only be reached from inside the sealed network via its Private Endpoint.
🗄️ What this means technically: the temporary data store (Cosmos DB) has "public access disabled" set in the platform's infrastructure code. It can only be reached through its Private Endpoint — an internal-only address. Even if someone knew the database address, they could not connect to it from outside the platform's network. This setting is enforced automatically every time the infrastructure is deployed (called IaC — Infrastructure as Code using Bicep).
✅
Marcus's entire journey stayed inside sealed corridors
From the moment Marcus's request passed the front door (Azure Front Door), it moved through private sealed connections (encrypted with TLS 1.2+) between each service. His data never crossed the public internet — even when travelling between platform services inside the VNet.
✅ What this means: every connection inside the platform — between gateway and connector, between connector and database — is encrypted and private. None of the platform's internal services have public internet addresses. If any one service were somehow compromised, the attacker would still be inside a sealed zone with no route to the others (called network segmentation via VNet subnets and NSGs).

Marcus clicks Submit — 8 gates in 200ms

When Marcus submits his business registration, his single click triggers 8 sequential security checks. The entire pipeline completes in under 200 milliseconds. Marcus never waits.

Marcus · Sole Proprietor · Sint Maarten
Gate 1 — Is Marcus actually who he says he is?. Gate 2 and 5 — Is Marcus allowed to do this, and is his request properly formed?.Gate 3 — Is Marcus making too many requests?.Gates 6, 7, 8 — Stamp a reference, prove our identity, send to the right place.
①
Gate 1 — Is Marcus actually who he says he is?
Marcus's login pass (JWT — JSON Web Token) is checked before anything else happens at Gate ① of the APIM security pipeline. If he is not logged in, or if the pass is fake or expired, the request stops here immediately — HTTP 401 is returned.
① What the platform does: it reads the login pass attached to the request, checks the digital stamp on it to confirm it was issued by the platform's own identity system and has not been tampered with (called JWT signature verification using RS256). It also checks that the pass has not expired (TTL check). If anything fails — wrong stamp, expired pass, missing pass — the request is rejected immediately and the registry is never contacted. The error returned is HTTP 401 (means "you are not logged in").
②
Gate 2 and 5 — Is Marcus allowed to do this, and is his request properly formed?
Even with a valid login pass (JWT), Marcus must have the right permission (called an OAuth scope) for the specific action he is requesting at Gate ②. And the data he sends must have exactly the right structure, checked at Gate ⑤ (payload schema validation).
② ⑤ What the platform does: Gate ② reads the list of permissions written into Marcus's login pass (called OAuth scopes). His pass says he can register a business — the permission is confirmed. Gate ⑤ checks that the data Marcus sent has all the required fields — business name, legal form, address — in the correct format. A missing field or wrong data type causes the request to be rejected (HTTP 403 means "not permitted", HTTP 400 means "badly formed request").
③⑤
Gate 3 — Is Marcus making too many requests?
The platform checks how many requests Marcus's session has made in the past minute at Gate ③ (rate limiting). This protects the platform from being overloaded — accidentally or deliberately — whether by a real citizen clicking too fast or by a malicious script.
③ What the platform does: each login session is allowed a sensible number of requests per minute (called rate limiting). The count is tracked per citizen session. Marcus has only made a few requests — well within the limit. If the limit is ever hit, the platform temporarily pauses further requests and tells the caller to try again shortly (HTTP 429 — "too many requests, please wait"). This also protects against automated scripts trying to extract large amounts of data.
⑥⑦⑧
Gates 6, 7, 8 — Stamp a reference, prove our identity, send to the right place
Before the request reaches the Business Registry, Gate ⑥ stamps a unique reference number (Tracking ID), Gate ⑦ proves the platform's own identity to COCI Portal (using Managed Identity — no password needed), and Gate ⑧ routes the request to the correct connector (REST · COCI Business Registry adapter).
⑥ ⑦ ⑧ What the platform does: Gate ⑥ generates a unique reference number (called a Tracking ID) and attaches it to Marcus's request — this reference will appear in every log entry for this action, making it possible to trace exactly what happened. Gate ⑦ proves that the request is coming from the authorised GoSXM platform — not from an unknown system (called Managed Identity authentication, no password needed). Gate ⑧ reads the request type and sends it to the correct connector — in this case, the Business Registry connector (REST · COCI).

Marcus logs in — what actually happens

Marcus taps Login. To him it looks like a simple redirect. Behind the scenes, three systems handshake in sequence to establish his identity — and no password ever touches the GoSXM platform.

Marcus · Sole Proprietor · Sint Maarten
Marcus taps Login — he is taken to the government's own login page. The government's eID system confirms Marcus is who he says he is.The platform converts the government confirmation into a session pass.Marcus's session pass is checked on every click — silently.
①②
Marcus taps Login — he is taken to the government's own login page
Marcus clicks Login on the GoSXM portal. He is immediately redirected to the national government's own identity verification page (the national eID system) — not a GoSXM page. His credentials are entered there, not here. The GoSXM portal never receives his password.
① ② What the platform does: the GoSXM portal sends Marcus's browser to the platform's identity service, which immediately redirects to the national eID login page. The GoSXM platform never receives Marcus's password or identity credentials — that responsibility stays entirely with the government's eID system. A secure handshake mechanism is used to prevent the login code being stolen in transit (called Authorization Code flow with PKCE — Proof Key for Code Exchange).
③
The government's eID system confirms Marcus is who he says he is
Marcus enters his eID credentials on the government's login page. The eID system verifies them and sends a digitally signed confirmation (called an id_token) back to the GoSXM platform's identity service (Entra External ID). Think of it as the government handing the platform a sealed, stamped letter confirming Marcus's identity.
③ What happens: the eID system sends a digitally signed identity confirmation (called an id_token) to the platform's identity service. The digital stamp on this confirmation (an RS256 cryptographic signature) was made using the eID system's private key — it cannot be faked. The platform verifies the stamp using a public list of the eID's verification keys (called the JWKS endpoint). If the stamp is valid, the confirmation is genuine.
④
The platform converts the government confirmation into a session pass
The platform's identity service (Entra External ID) receives the eID confirmation (id_token) and converts it into a short-lived session pass (JWT access_token, valid 2 hours) that the GoSXM platform understands. Marcus's citizen number (BSN) is scrambled into an unreadable code (HMAC-SHA256 hash) — the original number is thrown away and never stored anywhere.
④ What happens: the platform's identity service (called Entra External ID) maps the government's claims into platform claims. It runs Marcus's citizen number (BSN) through a one-way mathematical scramble (called HMAC-SHA256) to produce a scrambled citizen ID. The original number is discarded. The platform issues a short-lived login pass (called a JWT access_token, valid for 2 hours). This pass is stored on the platform's server — it is never placed inside Marcus's browser (no cookie, no local storage).
⑤
Marcus's session pass is checked on every click — silently
For every action Marcus takes — viewing his profile, submitting his business details, checking his tax status — the platform silently checks his session pass (JWT) at Gate ①. Near the end of its 2-hour life (TTL), the pass is renewed automatically using a refresh_token (a longer-lived renewal pass, 24-hour TTL) without interrupting Marcus.
⑤ What happens: every request Marcus makes includes his session pass (JWT). The platform checks the digital stamp and expiry time on every request at Gate ①. When the pass is close to expiring, the GoSXM server automatically exchanges it for a fresh one using a renewal pass (called a refresh_token, valid for 24 hours). Marcus never sees a re-login screen unless he has been inactive for more than 24 hours.

A legal agreement with Microsoft (called a Data Processing Agreement / DPA) covers government data handling requirements. Together these technical and legal measures ensure compliance with Sint Maarten government data requirements.
✓
If there is ever a security question, everything can be traced — without revealing Marcus's identity
The permanent audit log (Azure SQL — append-only table, 7-year retention) records every registry access using a scrambled citizen ID (HMAC-SHA256 hash of the BSN) and the unique Tracking ID. Investigators can trace exactly which registry was accessed, when, by which session, and the outcome — without the log containing any readable citizen personal data (PII — Personally Identifiable Information).
✓ How this works: the permanent audit log (stored in Azure SQL) records every access using a scrambled citizen ID (HMAC-SHA256 hash) instead of a real name or number. The table has a database-level guard (called a DDL trigger) that automatically blocks any attempt to delete or alter the table structure. No user role can UPDATE or DELETE existing rows — only INSERT is permitted. The unique Tracking ID links every entry back to a specific citizen request. Each entry is kept for 7 years and is accessible only to the compliance team (controlled by Row-Level Security).
